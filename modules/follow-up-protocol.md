# Follow-up Protocol (Canonical)

Follow-ups exist to **force clarity**, not to chase.

## Core rules
- **Max touches**: 3–4 touches, then disengage cleanly. Door stays open.
- **Each touch gets clearer**: Light → Structured → Decision point → (optional) Escalated.
- **Stop referencing missed messages** after disengaging. Re-enter later with something new and relevant.

## Default cadence

| Timing | Touch | Goal |
|--------|-------|------|
| Day 0 | Initial message | Make ask + context easy to respond to |
| Day 2–3 | Follow-up #1 (Light) | Surface it without pressure |
| Day 5–7 | Follow-up #2 (Structured) | Reinforce priority + request update |
| Day 10–14 | Follow-up #3 (Decision point) | Force a yes/no/when |
| After that | Close the loop | Disengage cleanly |

## Message patterns (copy/paste)

### Follow-up #1 (Light)
"Wanted to follow up in case it got buried."

### Follow-up #2 (Structured)
"Following up here. Would appreciate your thoughts when you have a moment."

### Follow-up #3 (Decision point)
"Let me know if this is something you can connect on, or if timing isn't right."

### Escalated (only if impact is meaningful)
"At this point, I need a clear update today so we can determine next steps."

### Close the loop
"No worries if timing isn't right. Happy to reconnect down the line."

## Two modes (choose intentionally)

### Relationship mode (network / maintain)
- Default to low pressure; optimize for long-term trust
- Use decision point to prevent awkward lingering

### Execution mode (projects / vendors / deliverables)
- Be explicit about **timing, blockers, next step**
- If needed, pair with tone ladder levels in `modules/edge-tone-ladder.md`
